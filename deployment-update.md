# Deployment Update 방식

## Strategy Type
> template의 내용이 변경되었을 경우 업데이트하는 방식이다.

### 1. Recreate
- **기존에 살아있던 모든 파드를 삭제** 후 신규 파드로 업데이트하는 방식
- **장점** : 신규 버전과 구버전이 명확히 구분이 가능함 (호환성 이슈 없음)
- **단점** : 기존 파드가 죽고 신규 파드가 기동되는 시간만큼 **다운타임(서비스 불가 시간)**이 발생함

### 2. RollingUpdate (기본값)
- **장점** : 구버전 파드를 하나씩 바꾸면서 배포하므로 **무중단 배포**가 가능
- **단점** :
    - 업데이트 기간 중 순간적으로 자원 사용량이 증가함 (기본 설정 시 최대 **125%**)
    - 신규 파드가 전부 교체될 때까지 **구버전과 신버전이 동시에 서비스에 노출**됨 (세션/데이터 호환성 고려 필요)

#### 2-1. maxUnavailable
- 업데이트 중 동시에 삭제(정지)될 수 있는 구버전 파드의 최대 개수/비율
- 기본값 : 25% (예: 파드가 4대면 업데이트 중 최소 3대는 살아있어야 함)

#### 2-2. maxSurge
- 업데이트 중 정해진 복제본(Replicas) 개수를 초과하여 생성될 수 있는 신규 버전 파드의 최대 개수/비율
- 기본값 : 25% (예: 파드가 4대면 업데이트 중 순간적으로 최대 5대까지 뜰 수 있음)

#### 2-3. 수치 조정 예시

##### 가정 1) Recreate 효과 내기
- `maxUnavailable: 100%` / `maxSurge: 0`
- 요약 : 기존 파드를 한 번에 다 삭제한 뒤, 신규 파드를 생성하라. (Recreate와 완전히 동일한 효과)

##### 가정 2) Blue/Green 형태 흉내 내기
- `maxUnavailable: 0%` / `maxSurge: 100%`
- 요약 : 기존 서비스를 한 대도 죽이지 않은 상태에서, 신규 버전 파드를 100% 다 띄운다. 신규 파드가 모두 Ready 상태가 되면 그때 기존 파드를 한 번에 지워나가는 방식. (순간 자원 사용량 200%까지 증가)

---

### Blue/Green 업데이트 방식
- 구버전(Blue)과 신버전(Green)의 전체 클러스터를 따로 띄워두고, 트래픽을 관장하는 Service 오브젝트의 라벨을 변경하여 한 번에 스위칭하는 방식
- **특징** : 업데이트 동안 두 버전이 동시에 호출되지 않음 (배포 순간 트래픽이 100% 전환)
- **단점** : 자원 사용량이 일시적으로 200%로 증가하여 비용 부담이 큼
- **구현** : Deployment의 자체 전략 속성에는 없으나, **K8s 기본 기능(Deployment 2개 + Service 1개 Selector 스위칭)**으로 구현 가능하며, 실무에서는 더 정교한 제어를 위해 **Argo Rollouts** 같은 외부 솔루션을 많이 도입함.
